1.0 General Security Concepts
1.1 Security Controls

Learning Objectives

Compare and contrast various types of security controls.

Security Controls Overview

Security Controls: Prevent security events, minimize impact and limit damage.

Control Categories vs Control Types

Categories Control Types
Technical Preventive
Managerial Deterrent
Operational Detective
Physical Corrective
Compensating

Control Categories

(i) Technical Controls

  • Implemented using a type of technical system
  • Operating system controls (allowing/disallowing functions)
  • Firewalls, anti-virus

(ii) Managerial Controls

  • Administrative controls associated with security design and implementation
  • Security policies, Standard Operating Procedures

(iii) Operational Controls

  • Controls implemented by people instead of systems
  • Security Guards, Awareness Programs (Posters, Best practices of Security)

(iv) Physical Controls

  • Limits physical access
  • Guard shack, Fences / Locks / Barricades, Badge Readers
Control Types

(i) Preventive Control Types

Function: Blocks access to a resource - "You shall not pass"

Examples:
  • Firewall Rules
  • Following Security Policies
  • Guard Shack Checks Identification
  • Door Locks

(ii) Deterrent Control Types

Function: Discourages an intrusion attempt - Does not directly prevent access - "Think Twice"

Examples:
  • Application Splash Screens
  • Threat of Demotion
  • Front Reception Desk
  • Warning Signs

(iii) Detective Control Types

Function: Identify and log an intrusion attempt - May not prevent access - "Find the Issue"

Examples:
  • Collecting reviewing system logs
  • Review Log-In Reports
  • Regular Patrol of Property
  • Motion Detectors

(iv) Corrective Security Control

Function: Occurs after the event has been detected - May reverse impact of event - Continue operating with minimal downtime - "Correct the Problem"

Examples:
  • Restoring from backups to mitigate ransomware infection
  • Create policies for reporting security issues
  • Contact law enforcement to manage criminal activity
  • Fire extinguisher

(v) Compensating Control Types

Function: Control using other means - Existing controls (other means) aren't sufficient - Temporary Basis - "Prevent the exploitation of a weakness"

Examples:
  • Firewall blocks a specific application instead of patching the app
  • Separation of duties, limit the scope of security concerns
  • Multiple security guards working at the same time (No complete access)
  • Backup Generator

(vi) Directive Control Types

Function: Direct a subject towards security compliance - Relatively weak security control - "Do this please"

Examples:
  • Store all sensitive files in a protected folder
  • Create compliance policies and procedures
  • Train users on proper security policy
  • Post a sign for "Authorized Personnel Only"

1.2 Fundamental Security Concepts
1.2.1 The CIA TRIAD

Learning Objectives

  • Explain the CIA triad and how the principles work together.
  • Identify technologies and processes that implement each principle.

Overview

CIA Triad: A foundational combination of principles that underpin information security: Confidentiality, Integrity, and Availability.

Fundamentals of Security – these principles are typically implemented together to protect data throughout its lifecycle.

CIA Triad Details

Confidentiality

  • Prevents disclosure of information to unauthorized individuals or systems.
  • Only the right people or systems should access specific information.
Examples:
  • Encryption (data-at-rest, in-transit, in-use)
  • Access controls and least privilege
  • Multi-factor authentication (MFA/2FA)

Integrity

  • Assures data is stored and transmitted exactly as intended.
  • Unauthorized modification can be detected and proven.
Examples:
  • Cryptographic hashes (e.g., SHA-256) to verify content
  • Digital signatures (asymmetric encryption over hashes)
  • X.509 certificates and code signing
  • Non-repudiation: proof that an action or origin is genuine

Availability

  • Ensures systems and data are accessible to authorized users when needed.
  • Designs emphasize resilience and minimal downtime.
Examples:
  • Redundancy and fault tolerance (clustering, failover)
  • Patching for stability and to close security holes
  • Backups and tested restore procedures
  • DDoS protection and capacity planning
1.2.2 Non-Repudiation

"You can't deny what you've said"

  • There’s no taking it back.
Example:
  • Signing a contract – your signature adds non-repudiation.
  • Proves you really signed the contract and others can verify it.

Perspective for cryptography:

  • Proof of integrity
  • Proof of origin, high assurance of authenticity

Proof of Integrity

  • Verify data does not change – remains accurate and consistent.

Cryptography: Use a hash to provide proof of integrity.

  • Represents data as a short string of text.
  • If the data changes, the hash changes.
  • Does not associate data with a particular individual.

Proof of Origin (Authentication)

  • Prove the message did not change (integrity) and came from the claimed source (authentication).

Cryptography: Digital Signatures

  • Private key is used to sign.
  • Message doesn’t need to be encrypted.
  • Only the owner has the private key; verification done with the public key.
  • Any change to the message invalidates the signature.

1.2.5 Zero Trust

Zero Trust

Many networks are relatively open internally. Once past the perimeter, few controls may exist. Zero Trust assumes nothing is trusted by default.

  • Covers every device, process, and person.
  • Everything is subject to verification: MFA, encryption, permissions, firewalls, monitoring and analysis.

Planes of Operation

Split a network into functional planes. These may be physical, virtual, or cloud components.

  • Data Plane: Processes frames/packets; forwarding, trunking, encrypting, NAT; firewalls.
  • Control Plane: Manages the data plane; defines policies and rules; routing/session/NAT tables; firewall rules.

Physical/SDN Architecture

Separate into functional tasks across planes.

Controlling Trust (Control Plane)

Adaptive Identity

  • Examine user identity and context; apply controls based on risk indicators.
  • Consider relationship to org, location, connection type, IP, requested resources; strengthen authentication.

Threat Scope Reduction

  • Limit access points and entry paths.

Policy-Driven Access Control

  • Combine adaptive identity with predefined policy rules.

Security Zones (Data Plane)

Where is someone connecting from and to? Security is more than a one-to-one relationship.

  • Trusted vs Untrusted; Internal vs External; various VPNs; departmental zones (Marketing, IT, HR).
  • Use zones to deny access based on rules (e.g., Untrusted → Trusted).
  • Some paths may be implicitly trusted (e.g., Trusted → Internal).

Policy Enforcement Point (PEP)

  • Subjects and systems: end users, apps, and non-human entities.
  • Gatekeeper: makes allow/deny decisions (abstracted set of devices).

Decision Flow

  • Policy Decision Point: overall process to make an authZ decision.
  • Policy Engine: evaluates requests against policy; returns Grant/Deny/Revoke.
  • Policy Administrator: communicates with PEP; issues tokens/credentials; instructs allow/deny.
1.2.6 Physical Security

Bollards

  • Prevent vehicle access; channel people through entry points.
  • Use in high-security areas; includes moats and concrete barriers.

Access Control Vestibule (Mantrap)

  • All doors normally unlocked: opening one locks the other.
  • All doors normally locked: unlocking one prevents the other from unlocking.
  • One-at-a-time or controlled groups to manage access through an area.

Fencing

  • Build a perimeter; robust and difficult to climb.
  • Increase height; use barbed wire to deter climbing.

Video Surveillance

  • CCTV reduces the need for physical guards; requires monitoring.
  • Motion recognition and object detection (e.g., license plates, faces).

Security Guards / Access Badges

  • Reception protection; validate employee identity.
  • Two-person control to minimize attack exposure.
  • Badges with photo/name; worn at all times; electronically logged.

Lighting / Sensors

  • More light → more security; improves visibility and facial recognition.
  • Infrared for low-light visibility and motion detection.
  • Pressure sensors for floors/windows; microwave and ultrasonic motion detection.
1.2.7 Detection and Disruption

Honeypots

  • Create a mock environment to attract and analyze attackers.
  • Often catches automated reconnaissance; observe attack methods.

Honeynets

  • Simulate real networks with multiple systems: servers, workstations, routers, firewalls.
  • Combine multiple honeypots for a believable infrastructure.

More info: projecthoneypot.org

Honeyfiles

  • Fake files/information (e.g., passwords.txt) placed to attract attackers.
  • Alerts trigger when accessed; acts as a trap.

Honeytokens

  • Track malicious actors with traceable data (API keys, fake emails).
  • Doesn’t provide real access; alerts when used or seen online.
  • Can include database records, cookies, tracking pixels, etc.
1.3 Change Management Process
1.3.1 Change Management

Learning Objectives

  • Understand the change management process and approval workflow.
  • Identify stakeholders and impact analysis requirements.

Overview

Change management ensures system updates are controlled, documented, and minimize risk to operations.

How to Make a Change

  • Upgrade software, patch applications, change firewall configuration, modify switch ports
  • Important to keep systems updated for security
  • Occur constantly - common risk for enterprises
Clear Policies: Frequency, duration, installation process, rollbacks

Change Approval Process

Formal process for managing change to avoid downtime, confusion, and mistakes.

  • Complete request forms with standard information
  • Determine purpose and scope of change
  • Schedule date/time and identify affected systems
  • Analyze risk and get approval from change control board
  • Get end-user acceptance after completion

Ownership

  • Individual/entity owns the process but doesn't perform actual changes
  • Manages process updates and ensures procedures are followed

Stakeholders

  • Identify who is impacted by the change
  • Research and identify all affected parties
  • Example: Software upgrade affects shipping, accounting, delivery, revenue

Impact Analysis

Determine risk value (high, medium, low) and potential consequences.

  • Risks: Fix doesn't work, breaks something else, OS failures, data corruption
  • Risks of not changing: Security vulnerabilities, application unavailability

Test Results

  • Use sandbox testing environment (isolated from production)
  • Test upgrades/patches before deployment
  • Confirm backout plan works

Backout Plan

  • Always have a way to revert changes
  • Prepare for the worst, hope for the best
  • Maintain good backups

Maintenance Window

  • Schedule changes during non-production hours
  • Consider overnight timing and seasonal factors
  • Most challenging part of the process

Standard Operating Procedure

  • Well-documented process available on intranet
  • Living document updated with process changes
  • Critical for organizational operations
1.3.2 Technical Change Management

Learning Objectives

  • Execute technical change management processes and procedures.
  • Manage dependencies, documentation, and version control.

Overview

Technical change management focuses on the "how" of implementing approved changes safely and efficiently.

Allow / Deny List

Security policies control application execution to prevent vulnerabilities and malware.

  • Allow List (Whitelist): Nothing runs unless approved - very restrictive
  • Deny List (Blacklist): Run any application except blocked ones

Restricted Activities

  • Change approval is specific to defined scope
  • Scope may need expansion during change window
  • Change management process determines next steps
  • Everyone understands the procedures

Downtime

  • Services may become unavailable during changes
  • Schedule during non-production hours when possible
  • Use secondary systems to minimize downtime
  • Automate processes and include in backout plan
  • Send notifications and calendar updates

Restarts

  • Common requirement for new configurations
  • Services: Stop and restart daemon/service
  • Applications: Close completely and launch new instance
  • Consider system recovery from power outages

Legacy Applications

  • Applications no longer supported by developer
  • Document the system thoroughly
  • Create specific processes and procedures
  • Address quirks and unknown behaviors

Dependencies

  • Complete A requires completing B first
  • Services may not start without other active services
  • Applications require specific library versions
  • Modifying one component may require changing others

Documentation

  • Keep documentation current with changes
  • Update diagrams for network modifications
  • Update policies/procedures for new systems
  • Require updates as part of change management

Version Control

  • Track changes to files/configurations over time
  • Easily revert to previous settings
  • Apply to router configs, OS patches, registry entries
  • May require additional management software
1.4 Appropriate uses of Cryptography
1.4.1 Public Key Infrastructure (PKI)

Learning Objectives

  • Understand PKI components and certificate lifecycle.
  • Compare symmetric and asymmetric encryption methods.

Overview

PKI encompasses policies, procedures, hardware, software, and people to create, distribute, manage, store, and revoke digital certificates. It binds public keys to people or devices through trusted Certificate Authorities.

Symmetric Encryption

  • Uses a single, shared key for both encryption and decryption
  • If compromised, requires a new key
  • Not scalable beyond ~10 people
Benefits: Very fast, less overhead than asymmetric, often combined with asymmetric encryption

Asymmetric Encryption

  • Uses two mathematically related keys created simultaneously
  • Private Key: Keep secret, only owner has access
  • Public Key: Can be shared with anyone
  • Private key is the only key that can decrypt data encrypted with the public key
  • Cannot derive private key from public key

Key Pair Generation

  • Build both public and private keys simultaneously
  • Uses randomization and large prime numbers
  • Everyone can have the public key, only owner has private key

Key Escrow

  • Third party holds decryption keys
  • May be within your organization or external
  • Legitimate business arrangement for data recovery
  • Government agencies may require access to partner data
  • Controversial but sometimes required
1.4.2 Encrypting Data

Learning Objectives

  • Identify appropriate encryption methods for different data types.
  • Understand data-at-rest and data-in-transit encryption.

Overview

Encryption protects data on storage devices (data-at-rest) and during transmission (data-in-transit) using various algorithms and key management techniques.

Data-at-Rest Encryption

  • Protect data on SSD, hard drives, USB drives, cloud storage
  • Full-disk encryption: BitLocker, FileVault
  • Single file encryption: EFS (Encrypting File System), third-party utilities
  • Built into NTFS - Properties → Encrypt Contents

Database Encryption

  • Transparent Encryption: Encrypt entire database with symmetric key
  • Record Level Encryption: Encrypt individual columns with separate keys
  • Non-sensitive data can remain in plaintext

Transport Encryption

  • Protect data traversing the network
  • Application-level: HTTPS in browsers
  • VPN: Client-based (SSL/TLS) or Site-to-Site (IPsec)
  • Encrypts all data regardless of application

Encryption Algorithms

  • Many ways to encrypt data - proper formula must be used
  • Both sides decide on algorithm before encryption
  • Consider security level, speed, implementation complexity

Cryptographic Keys

  • Algorithm is known - only the key is secret
  • Key determines encrypted data, hash value, digital signature
  • Key Lengths: Larger keys = more secure
  • Symmetric: 128-bit or larger
  • Asymmetric: 3,072-bit or larger

Key Stretching

  • Make weak keys stronger through multiple processes
  • Hash password → Hash the hash → Hash the hash of the hash
  • Makes brute force attacks require multiple decryption attempts
1.4.3 Key Exchange

Learning Objectives

  • Understand secure key exchange mechanisms.
  • Compare out-of-band vs in-band key exchange methods.

Overview

Key exchange solves the logistical challenge of sharing encryption keys across insecure mediums without physical transfer.

Key Exchange Methods

  • Out-of-Band: Not sent over network (telephone, courier, in-person)
  • In-Band: Sent over network, protected with additional encryption
  • Use asymmetric encryption to deliver symmetric keys

Real-time Encryption/Decryption

  • Share symmetric session key using asymmetric encryption
  • Client encrypts random symmetric key with server's public key
  • Server decrypts shared key and uses it for data encryption
  • Session keys must be changed often (ephemeral keys)
  • Must be unpredictable

Symmetric Key from Asymmetric Keys

  • Use public/private key cryptography to create symmetric key
  • Build same symmetric key on both sides without sending it
  • Key never transmitted across network
1.4.4 Encryption Technologies

Learning Objectives

  • Identify hardware and software encryption technologies.
  • Understand TPM, HSM, and key management systems.

Overview

Various hardware and software technologies provide cryptographic functions for single devices and large-scale environments.

Trusted Platform Module (TPM)

  • Provides encryption functions for a single device
  • Exists on motherboard, designed for cryptographic functions
  • Persistent Memory: Unique keys burned during manufacturing
  • Versatile Memory: Storage keys, BitLocker keys
  • Password protected - no dictionary attacks

Hardware Security Module (HSM)

  • Used in large environments with clusters and redundant power
  • Securely store thousands of cryptographic keys
  • High-end cryptographic hardware (plug-in card/separate device)
  • Key backup and secure storage
  • Cryptographic accelerators - offload CPU overhead

Key Management System

  • Centralized manager for all keys across services
  • Third-party software solution
  • Separate encryption keys from data
  • Create keys for specific services or cloud providers
  • Associate keys with end-users
  • Rotate keys on regular intervals
  • Log key use and events

Secure Enclave

  • Protected area for secrets - hardware processor
  • Isolated from main CPU
  • Used for fingerprints, FaceID, Apple Pay transactions
  • Secure boot verification, private keys & secrets
  • Own BOOT ROM, monitors system boot process
  • True random number generator, real-time memory encryption
  • Performs AES encryption in hardware
Memory Hook: A safe within your CPU
1.4.5 Obfuscation

Learning Objectives

  • Understand data obfuscation techniques.
  • Compare steganography, tokenization, and data masking.

Overview

Obfuscation makes information unclear or difficult to understand while hiding it in plain sight.

Steganography

  • Greek for "concealed writing"
  • Security through obscurity - "Hide the Key not the Lock"
  • If method is discovered, information can be recovered
  • Covertext: Container document or file

Common Steganography Techniques

  • Network-based: Embed messages in TCP packets
  • Image steganography: Embed message in image
  • Invisible watermarks: Yellow dots on printers
  • Audio steganography: Modify digital audio files
  • Video steganography: Sequence of images with secret information

Tokenization

  • Replace sensitive data with non-sensitive placeholder
  • Example: SSN → random number
  • Common in credit card processing (Apple Pay)
  • One-time use tokens
  • Original data and token aren't mathematically related

Data Masking

  • Hide some original data: **** **** **** 9989
  • Protects PII and sensitive data
  • May only hide from view - data intact in storage
  • Control view based on permissions
  • Techniques: Substituting, shuffling, encrypting, masking out
1.4.6 Hashing and Digital Signatures

Learning Objectives

  • Understand hashing algorithms and digital signatures.
  • Compare integrity verification methods and authentication.

Overview

Hashing provides integrity verification while digital signatures provide authentication, non-repudiation, and integrity.

Hashes

  • Represent data as short string of text (message digest/fingerprint)
  • One-way trip: NOT encryption, cannot recreate data
  • Used for password storage (confidentiality)
  • Verify downloaded documents match original (integrity)
  • Can create digital signatures (authentication, non-repudiation)

Practical Hashing

  • File verification: Compare downloaded hash with posted hash
  • Password storage: Store salted hash instead of password
  • Compare hashes during authentication

Salt

  • Random data added to password when hashing
  • Makes brute force more difficult
  • Every user gets different random salt
  • Same passwords → Different hashes
  • Prevents rainbow table attacks

Hash Collisions

  • Different inputs creating same hash (extremely rare)
  • Don't use MD5 - collisions are possible
  • Use SHA256 (256 bits/64 hexadecimal characters)
  • One minor change causes completely different output

Digital Signatures

  • Prove message wasn't changed (integrity)
  • Prove source of message (authentication)
  • Ensure signature isn't fake (non-repudiation)
  • Sign with private key, verify with public key
  • Any message change invalidates signature

Creating a Digital Signature

  • Email client hashes plaintext
  • Encrypts hash with private key
  • Adds digital signature to plaintext (no encryption)
  • Only sender has private key

Verifying a Digital Signature

  • Decrypt digital signature using sender's public key
  • Perform hashing algorithm on received plaintext
  • Compare hashes
  • If match: Document is same as sent, confirmed from sender
1.4.7 Blockchain Technologies

Learning Objectives

  • Understand blockchain technology and distributed ledgers.
  • Explain blockchain process and tamper-resistance mechanisms.

Overview

Blockchain is a tamper-resistant digital notebook shared across every computer in a network, used to track transactions without central authority.

What is Blockchain?

  • A distributed ledger - shared and copied to every computer in network
  • Keeps track of transactions like money transfers, votes, ownership records
  • No central authority needed
  • Everyone on the network maintains the ledger
  • Records and replicates to anyone and everyone
Memory Hook: Imagine a classroom where every student keeps a copy of the gradebook. If someone tries to sneak in a higher grade, they'd be caught because their gradebook wouldn't match everyone else's. That's blockchain — everyone has a copy of the truth.

Blockchain Applications

  • Payment processing
  • Digital identification
  • Digital voting
  • Transferring cryptocurrency (BTC)
  • Transferring house titles

Blockchain Process

  1. Transaction sent to every computer/node in decentralized network for verification
  2. Verified transaction added to new block of data with other recent transactions
  3. Secure code (hash) calculated from previous transaction data blocks added to new block
  4. Block added to end of blockchain, added to everyone's ledger

Why the Hash Matters

  • If even 1 character in a block changes, the hash changes completely
  • This breaks the chain, making tampering immediately obvious
  • Network rejects altered chains
  • All following hashes are automatically recalculated
Memory Hook: Think of the hash like a tamper-evident seal. If the seal's broken, everyone knows it's been messed with.

Why Is Blockchain Powerful?

  • Tamper-resistant: Can't change past transactions without changing every copy (practically impossible)
  • Decentralized: No single person or company owns the data
  • Trustless: Don't need to trust middlemen like banks or notaries
1.4.8 Certificates

Learning Objectives

  • Understand digital certificates and certificate authorities.
  • Compare public vs private certificate authorities and revocation methods.

Overview

Digital certificates bind public keys with digital signatures to establish trust in digital communications through Certificate Authorities.

Digital Certificates

  • Public Key Certificates: Bind a public key with a digital signature
  • Include other details about key holder
  • Digital signatures add trust through Certificate Authorities
  • Web of Trust: Additional users for additional trust ("A friend of a friend we can trust")
  • Certificates within organizations may not need 3rd party CA
  • Certificate creation can be built into the OS

Standard Format: X.509

  • Certificate Details: Serial Number, Version, Signature Algorithm
  • Issuer, Name of Certificate Holder, Public Key
  • Standardized format for digital certificates

Root of Trust

  • Everything in IT security requires trust (fundamental characteristic)
  • How can we build trust from something unknown?
  • Have a 3rd party vouch for this connection
  • Root of Trust: Inherently and implicitly trusted component
  • Can be hardware, software, or other component (HSM, Secure Enclave, etc.)

Certificate Authorities (CA)

  • When connecting to random websites, how can you trust them?
  • CA has digitally signed the website certificate
  • Browser trusts the CA, you trust the CA
  • Real-time verification process

Third-Party CAs

  • Built into your browser
  • Hundreds of certificate authorities that are trusted
  • Companies pay the CA and become trusted by your browser
  • Charge comes from authority signing the certification and providing validation

Certificate Signing Requests (CSR)

  • Create a key pair, then send the public key to the CA to be signed
  • CA validates the request (confirms DNS emails and website ownership)
  • Once approved, signed with CA's private key and sent back

Private Certificate Authorities

  • You are your own CA - build it in-house
  • Your devices must trust the internal CA
  • Take public certificate → Install on everyone's computer in organization
  • Everyone's machine trusts the internal CA same as external CA
  • Software Packages: Windows Certificate Services, Open CA

Self-Signed Certificates

  • Internal certificates don't need to be signed by Public CA
  • Company is the only one going to use it
  • No need to purchase trust for devices that trust you
  • Build your own CA and issue your own certificates
  • Install the CA certificate/trusted chain on all devices
  • All devices know and inherently trust because of the internal CA

Wildcard Certificates

  • Subject Alternative Name (SAN): Extension to X.509 Certificate
  • Allows certificate to support various domains
  • Wildcard Domain: Certificates based on server name
  • Wildcard domain applies to all server names with associated domain name

Key Revocation

  • Certificate Revocation List (CRL): Maintained by CA
  • Contains many revocations in a large file
  • Many different ways to revoke a certificate
  • Example - Heartbleed: Vulnerability in OpenSSL that put private keys at risk
  • OpenSSL patched, web server certificates were replaced

OCSP Stapling

Problem: Using CRL is slow and bulky. If 1000 people visit a website, they all have to download the CRL, search for the certificate, and confirm it's valid. That doesn't scale well.
  • Online Certificate Status Protocol (OCSP): Provides scalability for OCSP checks
  • CA responsible for responding to all client OCSP requests (inefficient)
  • Instead, have certificate holder verify their own status
  • Status information stored on certificate holder's server
  • OCSP status "stapled" into SSL/TLS handshake, digitally signed by CA
  • Browser can handle all checks for revocation
  • Alternative: 3rd party server to provide OCSP responder via HTTP
2.0 Threats, Vulnerabilities, and Mitigations
2.1 Threat Actors

Learning Objectives

  • Identify different types of threat actors and their characteristics.
  • Understand threat actor motivations and capabilities.

Overview

Threat actors are entities responsible for events that impact the safety of other entities. Understanding their attributes, motivations, and capabilities is crucial for cybersecurity defense.

Threat Actor Definition

  • Entity responsible for events that impact safety of another entity
  • Malicious actors with various motivations and capabilities

Threat Actor Attributes

  • Internal/External: Location relative to target organization
  • Resources/Funding: From no money/resources to extensive funding
  • Level of Sophistication: Do they know what they're doing?

Motivations of Threat Actors

  • Data exfiltration
  • Espionage
  • Service disruption
  • Blackmail
  • Ethical reasons
  • Revenge

Nation State

  • Type: External entity (Government and national security)
  • Motivations: Data exfiltration, war, various government objectives
  • Characteristics: Constant attacks, massive resources
  • Advanced Persistent Threat (APT): Long-term, sophisticated campaigns
  • Highest Sophistication: Resources of entire government, military control, utilities, finances

Unskilled Attackers

  • Behavior: Run pre-made scripts without knowledge
  • Type: Usually external
  • Characteristics: Unsophisticated, not many resources
  • Accessibility: Anyone can do this

Hacktivist

  • Definition: Hacker with purpose (philosophy, revenge, disruption)
  • Type: Often external entity (could infiltrate and become insider)
  • Sophistication: Very sophisticated with specific hacks
  • Methods: DoS, defacing, private document release
  • Resources: Limited funding

Insider Threat

  • Definition: More than passwords on sticky notes
  • Motivations: Revenge, financial gain
  • Resources: Extensive (exist within organization)
  • Advantages: Taking advantage of existing resources, hiring process
  • Type: Internal entity

Organized Crime

  • Definition: Professional criminals motivated by money
  • Type: Almost always external
  • Purpose: Everything is for profit
  • Sophistication: Very sophisticated (best hacking money can buy)
  • Structure: Crime that's organized with division of labor

Shadow IT

  • Definition: Going rogue, working around internal IT organization
  • Behavior: Builds own infrastructure, uses cloud
  • Characteristics: Unencumbered, limited resources
  • Funding: Company budget
2.2 Attack Vectors

Learning Objectives

  • Identify common attack vectors and methods used by threat actors.
  • Understand various delivery mechanisms for malicious activities.

Overview

Attack vectors are methods used by attackers to gain access or infect targets. Understanding these vectors helps in developing appropriate defenses.

Message-Based Vectors

  • Most people use messaging systems - very successful vector
  • Examples: Malicious links in email, SMS attacks
  • Phishing Attacks: People want to click links
  • Deliver malware to users through social engineering

Image-Based Vectors

  • Images more difficult to identify than text-based threats
  • SVG Format: Scalable Vector Graphics (XML file with embedded information)
  • Security Concerns: HTML injection, JavaScript attack code, XSS
  • Browsers must provide input validation to avoid running malicious code

File-Based Vectors

  • More than just executables - malicious code can be anywhere
  • Examples: Adobe PDF, ZIP/RAR files, Microsoft Office
  • File formats containing other objects

Voice-Call Vectors

  • Vishing: Voice phishing
  • Spam over IP: VoIP-based spam
  • War Dialing: Unpublished phone numbers with system access
  • Call Tampering: Denial of Service attacks

Removable Device Vectors

  • Gets around firewall through USB interface
  • Malicious software on USB flash drives
  • Can infect air-gapped networks
  • Industrial systems, high-security services
  • USB devices can act as keyboards (hacker on a chip)
  • Data exfiltration - terabyte of data, zero bandwidth used

Vulnerable Software Vectors

  • Client-based: Infected executables, known/unknown vulnerabilities
  • May require constant patching (updates)
  • Agentless Systems: No installed executable
  • Compromised software on server affects users
  • Client runs new instance each time

Unsupported System Vectors

  • Patching is important security tool with ongoing fixes
  • Unsupported systems aren't patched
  • Outdated operating systems
  • Single system could be entry point

Unsecure Network Vectors

  • Networks connect everything - ease of access for attackers
  • View all non-encrypted data
  • Wireless: Outdated protocols (WEP, WPA, WPA2), rogue networks
  • Wired: Unsecure interfaces, Bluetooth

Open Service Ports

  • Most network services connect over TCP or UDP ports
  • Every open port is opportunity for attacker
  • Every application has own port number
  • More services = More open ports = Less secure system
  • Port-based firewall rules allow traffic to open ports

Default Credentials

  • Many devices still have default usernames and passwords
  • Change them!
  • Many devices require password change on first login
  • Administrator access taken away
  • Very easy to find default credentials

Supply Chain Vectors

  • Tamper with underlying infrastructure
  • Added during or after manufacturing process (3rd party)
  • Managed Service Providers (MSP): Access many customer networks from one location
  • Great place to start (centralized)
  • Suppliers: Counterfeit networking equipment
  • Example: 2020 Fake Cisco Catalyst switches
2.2.1 Phishing

Learning Objectives

  • Understand phishing techniques and prevention methods.
  • Identify various types of phishing attacks.

Overview

Phishing combines social engineering with spoofing, delivered through email or text to trick users into clicking malicious links.

What is Phishing?

  • Social engineering combined with spoofing
  • Delivered by email, text
  • Try to get users to click on resources (links) with malicious intent

Risks of Phishing

  • Using login credentials on fake website
  • Exposing sensitive information

Best Practices Against Phishing

  • Don't click links!
  • Verify everything
  • Something just isn't right

Business Email Compromise

  1. Email Spoofing: Attackers take advantage of trust given to emails
    • Verify emails for spoofing, similar at first glance
  2. Financial Fraud:
    • Send emails with updated bank information
    • Modify wire transfer details
  3. Malware Delivery: Clicking on link with malware

Tricks and Misdirection

  • Digital Sleight of Hand: How they're so successful
  • Typosquatting: Destination URL is not legitimate domain name (misdirection)
  • Pretexting: Just lying to get information
    • Attacker is character in situation they created
    • Gaslighting techniques

Various Bait

  • Voice Phishing:
    • Caller ID spoofing
    • Fake security automation
  • Smishing (SMS Phishing):
    • Done by text
    • Spoofing techniques
