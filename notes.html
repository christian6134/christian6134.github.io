1.0 General Security Concepts
1.1 Security Controls

Learning Objectives

Compare and contrast various types of security controls.

Security Controls Overview

Security Controls: Prevent security events, minimize impact and limit damage.

Control Categories vs Control Types

Categories Control Types

Control Categories

(i) Technical Controls

  • Implemented using a type of technical system
  • Operating system controls (allowing/disallowing functions)
  • Firewalls, anti-virus

(ii) Managerial Controls

  • Administrative controls associated with security design and implementation
  • Security policies, Standard Operating Procedures

(iii) Operational Controls

  • Controls implemented by people instead of systems
  • Security Guards, Awareness Programs (Posters, Best practices of Security)

(iv) Physical Controls

  • Limits physical access
  • Guard shack, Fences / Locks / Barricades, Badge Readers
Control Types

(i) Preventive Control Types

Function: Blocks access to a resource - "You shall not pass"

Examples:
  • Firewall Rules
  • Following Security Policies
  • Guard Shack Checks Identification
  • Door Locks

(ii) Deterrent Control Types

Function: Discourages an intrusion attempt - Does not directly prevent access - "Think Twice"

Examples:
  • Application Splash Screens
  • Threat of Demotion
  • Front Reception Desk
  • Warning Signs

(iii) Detective Control Types

Function: Identify and log an intrusion attempt - May not prevent access - "Find the Issue"

Examples:
  • Collecting reviewing system logs
  • Review Log-In Reports
  • Regular Patrol of Property
  • Motion Detectors

(iv) Corrective Security Control

Function: Occurs after the event has been detected - May reverse impact of event - Continue operating with minimal downtime - "Correct the Problem"

Examples:
  • Restoring from backups to mitigate ransomware infection
  • Create policies for reporting security issues
  • Contact law enforcement to manage criminal activity
  • Fire extinguisher

(v) Compensating Control Types

Function: Control using other means - Existing controls (other means) aren't sufficient - Temporary Basis - "Prevent the exploitation of a weakness"

Examples:
  • Firewall blocks a specific application instead of patching the app
  • Separation of duties, limit the scope of security concerns
  • Multiple security guards working at the same time (No complete access)
  • Backup Generator

(vi) Directive Control Types

Function: Direct a subject towards security compliance - Relatively weak security control - "Do this please"

Examples:
  • Store all sensitive files in a protected folder
  • Create compliance policies and procedures
  • Train users on proper security policy
  • Post a sign for "Authorized Personnel Only"

1.2 Fundamental Security Concepts
1.2.1 The CIA TRIAD

Learning Objectives

  • Explain the CIA triad and how the principles work together.
  • Identify technologies and processes that implement each principle.

Overview

CIA Triad: A foundational combination of principles that underpin information security: Confidentiality, Integrity, and Availability.

Fundamentals of Security – these principles are typically implemented together to protect data throughout its lifecycle.

CIA Triad Details

Confidentiality

  • Prevents disclosure of information to unauthorized individuals or systems.
  • Only the right people or systems should access specific information.
Examples:
  • Encryption (data-at-rest, in-transit, in-use)
  • Access controls and least privilege
  • Multi-factor authentication (MFA/2FA)

Integrity

  • Assures data is stored and transmitted exactly as intended.
  • Unauthorized modification can be detected and proven.
Examples:
  • Cryptographic hashes (e.g., SHA-256) to verify content
  • Digital signatures (asymmetric encryption over hashes)
  • X.509 certificates and code signing
  • Non-repudiation: proof that an action or origin is genuine

Availability

  • Ensures systems and data are accessible to authorized users when needed.
  • Designs emphasize resilience and minimal downtime.
Examples:
  • Redundancy and fault tolerance (clustering, failover)
  • Patching for stability and to close security holes
  • Backups and tested restore procedures
  • DDoS protection and capacity planning
1.2.2 Non-Repudiation

"You can't deny what you've said"

  • There’s no taking it back.
Example:
  • Signing a contract – your signature adds non-repudiation.
  • Proves you really signed the contract and others can verify it.

Perspective for cryptography:

  • Proof of integrity
  • Proof of origin, high assurance of authenticity

Proof of Integrity

  • Verify data does not change – remains accurate and consistent.

Cryptography: Use a hash to provide proof of integrity.

  • Represents data as a short string of text.
  • If the data changes, the hash changes.
  • Does not associate data with a particular individual.

Proof of Origin (Authentication)

  • Prove the message did not change (integrity) and came from the claimed source (authentication).

Cryptography: Digital Signatures

  • Private key is used to sign.
  • Message doesn’t need to be encrypted.
  • Only the owner has the private key; verification done with the public key.
  • Any change to the message invalidates the signature.

1.2.5 Zero Trust

Zero Trust

Many networks are relatively open internally. Once past the perimeter, few controls may exist. Zero Trust assumes nothing is trusted by default.

  • Covers every device, process, and person.
  • Everything is subject to verification: MFA, encryption, permissions, firewalls, monitoring and analysis.

Planes of Operation

Split a network into functional planes. These may be physical, virtual, or cloud components.

  • Data Plane: Processes frames/packets; forwarding, trunking, encrypting, NAT; firewalls.
  • Control Plane: Manages the data plane; defines policies and rules; routing/session/NAT tables; firewall rules.

Physical/SDN Architecture

Separate into functional tasks across planes.

Controlling Trust (Control Plane)

Adaptive Identity

  • Examine user identity and context; apply controls based on risk indicators.
  • Consider relationship to org, location, connection type, IP, requested resources; strengthen authentication.

Threat Scope Reduction

  • Limit access points and entry paths.

Policy-Driven Access Control

  • Combine adaptive identity with predefined policy rules.

Security Zones (Data Plane)

Where is someone connecting from and to? Security is more than a one-to-one relationship.

  • Trusted vs Untrusted; Internal vs External; various VPNs; departmental zones (Marketing, IT, HR).
  • Use zones to deny access based on rules (e.g., Untrusted → Trusted).
  • Some paths may be implicitly trusted (e.g., Trusted → Internal).

Policy Enforcement Point (PEP)

  • Subjects and systems: end users, apps, and non-human entities.
  • Gatekeeper: makes allow/deny decisions (abstracted set of devices).

Decision Flow

  • Policy Decision Point: overall process to make an authZ decision.
  • Policy Engine: evaluates requests against policy; returns Grant/Deny/Revoke.
  • Policy Administrator: communicates with PEP; issues tokens/credentials; instructs allow/deny.
1.2.6 Physical Security

Bollards

  • Prevent vehicle access; channel people through entry points.
  • Use in high-security areas; includes moats and concrete barriers.

Access Control Vestibule (Mantrap)

  • All doors normally unlocked: opening one locks the other.
  • All doors normally locked: unlocking one prevents the other from unlocking.
  • One-at-a-time or controlled groups to manage access through an area.

Fencing

  • Build a perimeter; robust and difficult to climb.
  • Increase height; use barbed wire to deter climbing.

Video Surveillance

  • CCTV reduces the need for physical guards; requires monitoring.
  • Motion recognition and object detection (e.g., license plates, faces).

Security Guards / Access Badges

  • Reception protection; validate employee identity.
  • Two-person control to minimize attack exposure.
  • Badges with photo/name; worn at all times; electronically logged.

Lighting / Sensors

  • More light → more security; improves visibility and facial recognition.
  • Infrared for low-light visibility and motion detection.
  • Pressure sensors for floors/windows; microwave and ultrasonic motion detection.
1.2.7 Detection and Disruption

Honeypots

  • Create a mock environment to attract and analyze attackers.
  • Often catches automated reconnaissance; observe attack methods.

Honeynets

  • Simulate real networks with multiple systems: servers, workstations, routers, firewalls.
  • Combine multiple honeypots for a believable infrastructure.

More info: projecthoneypot.org

Honeyfiles

  • Fake files/information (e.g., passwords.txt) placed to attract attackers.
  • Alerts trigger when accessed; acts as a trap.

Honeytokens

  • Track malicious actors with traceable data (API keys, fake emails).
  • Doesn’t provide real access; alerts when used or seen online.
  • Can include database records, cookies, tracking pixels, etc.
1.3 Change Management Process
1.3.1 Change Management

Learning Objectives

  • Understand the change management process and approval workflow.
  • Identify stakeholders and impact analysis requirements.

Overview

Change management ensures system updates are controlled, documented, and minimize risk to operations.

How to Make a Change

  • Upgrade software, patch applications, change firewall configuration, modify switch ports
  • Important to keep systems updated for security
  • Occur constantly - common risk for enterprises
Clear Policies: Frequency, duration, installation process, rollbacks

Change Approval Process

Formal process for managing change to avoid downtime, confusion, and mistakes.

  • Complete request forms with standard information
  • Determine purpose and scope of change
  • Schedule date/time and identify affected systems
  • Analyze risk and get approval from change control board
  • Get end-user acceptance after completion

Ownership

  • Individual/entity owns the process but doesn't perform actual changes
  • Manages process updates and ensures procedures are followed

Stakeholders

  • Identify who is impacted by the change
  • Research and identify all affected parties
  • Example: Software upgrade affects shipping, accounting, delivery, revenue

Impact Analysis

Determine risk value (high, medium, low) and potential consequences.

  • Risks: Fix doesn't work, breaks something else, OS failures, data corruption
  • Risks of not changing: Security vulnerabilities, application unavailability

Test Results

  • Use sandbox testing environment (isolated from production)
  • Test upgrades/patches before deployment
  • Confirm backout plan works

Backout Plan

  • Always have a way to revert changes
  • Prepare for the worst, hope for the best
  • Maintain good backups

Maintenance Window

  • Schedule changes during non-production hours
  • Consider overnight timing and seasonal factors
  • Most challenging part of the process

Standard Operating Procedure

  • Well-documented process available on intranet
  • Living document updated with process changes
  • Critical for organizational operations
1.3.2 Technical Change Management

Learning Objectives

  • Execute technical change management processes and procedures.
  • Manage dependencies, documentation, and version control.

Overview

Technical change management focuses on the "how" of implementing approved changes safely and efficiently.

Allow / Deny List

Security policies control application execution to prevent vulnerabilities and malware.

  • Allow List (Whitelist): Nothing runs unless approved - very restrictive
  • Deny List (Blacklist): Run any application except blocked ones

Restricted Activities

  • Change approval is specific to defined scope
  • Scope may need expansion during change window
  • Change management process determines next steps
  • Everyone understands the procedures

Downtime

  • Services may become unavailable during changes
  • Schedule during non-production hours when possible
  • Use secondary systems to minimize downtime
  • Automate processes and include in backout plan
  • Send notifications and calendar updates

Restarts

  • Common requirement for new configurations
  • Services: Stop and restart daemon/service
  • Applications: Close completely and launch new instance
  • Consider system recovery from power outages

Legacy Applications

  • Applications no longer supported by developer
  • Document the system thoroughly
  • Create specific processes and procedures
  • Address quirks and unknown behaviors

Dependencies

  • Complete A requires completing B first
  • Services may not start without other active services
  • Applications require specific library versions
  • Modifying one component may require changing others

Documentation

  • Keep documentation current with changes
  • Update diagrams for network modifications
  • Update policies/procedures for new systems
  • Require updates as part of change management

Version Control

  • Track changes to files/configurations over time
  • Easily revert to previous settings
  • Apply to router configs, OS patches, registry entries
  • May require additional management software
